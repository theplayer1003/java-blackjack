# 오브젝트 (조영호) 가이드라인

> 코드 리뷰 시 참고할 「오브젝트 — 코드로 이해하는 객체지향 설계」의 핵심 원칙들을 정리한 문서입니다.

---

## 1장. 객체, 설계

### 1.1 티켓 판매 애플리케이션 — 핵심 교훈

- 소프트웨어 모듈의 세 가지 목적 (로버트 마틴):
    1. 실행 중에 제대로 동작한다.
    2. 변경을 위해 존재한다.
    3. 코드를 읽는 사람과 의사소통한다.
- 모든 모듈은 제대로 동작해야 하고, 변경이 용이해야 하며, 이해하기 쉬워야 한다.

### 1.2 무엇이 문제인가

- **예상을 빗나가는 코드**: 코드를 이해하기 위해 여러 가지 세부 사항을 기억하고 있어야 한다면 그 코드는 좋지 않다.
- **변경에 취약한 코드**: 한 클래스가 다른 클래스의 내부에 대해 더 많이 알수록 변경하기 어려워진다.
- 이것이 객체 사이의 **의존성(dependency)** 문제다.
- 의존성을 완전히 없애는 것이 아니라, 애플리케이션 기능 구현에 필요한 **최소한의 의존성**만 유지한다.

### 1.3 해결 방법: 자율적인 객체

- 각 객체가 자신의 데이터를 **스스로 처리**하도록 한다.
- 개념적이나 물리적으로 객체 내부의 세부적인 사항을 감추는 것이 **캡슐화**이다.
- 캡슐화의 목적은 변경하기 쉬운 객체를 만드는 것이다.
- **인터페이스와 구현의 분리**: 외부에서는 인터페이스에만 의존하고, 구현의 세부사항은 내부에 감춘다.

## 2장. 객체지향 프로그래밍

### 2.1 협력, 객체, 클래스

- 객체지향은 **클래스가 아니라 객체에 초점**을 맞춰야 한다.
- 어떤 클래스가 필요한지가 아니라, **어떤 객체들이 필요한지** 고민한다.
- 객체를 독립적인 존재가 아니라 기능을 구현하기 위해 **협력하는 공동체**의 일원으로 본다.

### 2.2 도메인 개념을 따르는 프로그램 구조

- 클래스의 이름은 대응하는 **도메인 개념**의 이름과 동일하거나 적어도 유사해야 한다.
- 클래스 사이의 관계도 최대한 도메인 개념 사이에 맺어진 관계와 유사하게 만든다.

### 2.3 다형성

- **컴파일 시간 의존성과 실행 시간 의존성이 다를 수 있다.**
- 코드의 의존성과 실행 시점의 의존성이 다르면 다르를수록 코드를 이해하기 어려워지지만, 코드는 더 유연해지고 확장 가능해진다.
- 설계가 유연해질수록 코드를 이해하고 디버깅하기는 더 어려워진다. 이 트레이드오프를 인식해야 한다.

### 2.4 상속과 다형성

- 업캐스팅: 자식 클래스가 부모 클래스를 대신하는 것.
- **동적 바인딩(dynamic binding)**: 메시지를 수신한 객체의 실제 타입에 따라 실행되는 메서드가 결정된다.
- 구현 상속(서브클래싱)과 인터페이스 상속(서브타이핑)을 구분한다.

### 2.5 추상화의 힘

- 추상화를 사용하면 세부적인 내용을 무시한 채 상위 정책을 쉽고 간단하게 표현할 수 있다.
- 추상화를 이용하면 상위 정책을 표현하면서도 불필요한 세부 사항을 생략할 수 있다.
- 추상화를 이용해 상위 정책을 기술한다는 것은 기본적인 애플리케이션의 협력 흐름을 기술한다는 것을 의미한다.

## 3장. 역할, 책임, 협력

### 3.1 협력 (Collaboration)

- 객체지향 시스템은 자율적인 객체들의 공동체다.
- 객체가 다른 객체에게 도움을 요청하는 것이 **메시지 전송**이다.
- 두 객체 사이의 협력은 하나의 객체가 다른 객체에게 도움을 요청할 때 시작된다.
- **협력이 설계를 위한 문맥(context)을 결정한다.**

### 3.2 책임 (Responsibility)

- 협력에 참여하기 위해 객체가 수행하는 로직을 **책임**이라고 부른다.
- 책임은 크게 **하는 것(doing)**과 **아는 것(knowing)**으로 나뉜다.
    - **하는 것**: 객체를 생성하거나 계산을 수행하는 등 스스로 하는 것, 다른 객체의 활동을 제어하고 조절하는 것
    - **아는 것**: 사적인 정보에 관해 아는 것, 관련된 객체에 관해 아는 것, 자신이 유도하거나 계산할 수 있는 것에 관해 아는 것
- **책임이 객체지향 설계의 핵심이다.**
- CRC 카드: 후보(Candidate), 책임(Responsibility), 협력자(Collaborator)

### 3.3 책임 할당

- **정보 전문가 패턴 (INFORMATION EXPERT)**: 책임을 수행하는 데 필요한 정보를 가장 많이 알고 있는 객체에게 책임을 할당한다.
- 객체에게 책임을 할당하기 위해서는 먼저 **협력이라는 문맥**을 정의한다.
- 협력에 필요한 메시지를 찾고, 메시지에 적합한 객체를 선택한다.

### 3.4 역할 (Role)

- 객체가 특정한 협력 안에서 수행하는 책임의 집합을 **역할**이라고 부른다.
- 역할은 재사용 가능하고 유연한 객체지향 설계를 낳는 매우 중요한 구성요소다.
- 역할을 구현하는 가장 일반적인 방법은 **추상 클래스**와 **인터페이스**를 사용하는 것이다.

## 4장. 설계 품질과 트레이드오프

### 4.1 데이터 중심 설계 vs 책임 중심 설계

- **데이터 중심 설계**: 객체가 포함해야 하는 데이터에 초점. 객체의 상태에 초점.
    - 결과: 캡슐화 위반, 높은 결합도, 낮은 응집도
- **책임 중심 설계**: 다른 객체에게 제공해야 하는 행위에 초점. 객체의 행동에 초점.
    - 결과: 높은 캡슐화, 낮은 결합도, 높은 응집도
- **"이 객체가 어떤 데이터를 가져야 하는가?"가 아니라 "이 객체가 어떤 책임을 져야 하는가?"를 먼저 결정한다.**

### 4.2 캡슐화 위반의 징후

- getter/setter를 통해 내부 상태를 그대로 노출하면 캡슐화가 아니다.
- 접근자와 수정자에 과도하게 의존하는 설계 방식을 **추측에 의한 설계 전략(design-by-guessing strategy)**이라고 한다.
- 내부 구현의 변경이 외부로 퍼져나가는 **파급 효과(ripple effect)**를 최소화한다.

### 4.3 높은 결합도의 문제

- 결합도가 높으면 한 객체를 변경할 때 다른 객체도 함께 변경해야 한다.
- **직접적인 결합**: 한 객체가 다른 객체의 구체적인 구현에 의존
- **간접적인 결합**: 한 객체가 다른 객체를 통해 제3의 객체에 의존 (데이터 중심 설계에서 흔함)

### 4.4 낮은 응집도의 문제

- 서로 연관성 없는 기능이나 데이터가 하나의 클래스 안에 뭉쳐 있으면 응집도가 낮다.
- 응집도가 낮으면 변경의 이유가 여러 가지가 된다.
- 하나의 요구사항 변경으로 인해 동시에 여러 모듈을 수정해야 한다면 응집도가 낮은 것이다.

### 4.5 자율적인 객체를 향해

- **스스로 자신의 데이터를 책임지는 객체**: 객체가 어떤 데이터를 가지고 있는지와 그 데이터에 대해 수행할 수 있는 오퍼레이션이 무엇인지를 함께 고려한다.
- "이 객체가 어떤 데이터를 포함해야 하는가?"에 이어 "이 데이터에 대해 수행할 수 있는 오퍼레이션은 무엇인가?"를 묻는다.

## 5장. 책임 할당하기 — GRASP 패턴

### 5.1 INFORMATION EXPERT (정보 전문가)

- 책임을 수행할 정보를 알고 있는 객체에게 책임을 할당한다.
- "이 책임을 수행하기 위해 필요한 정보를 가장 많이 아는 객체는 누구인가?"

```
예: 블랙잭에서 "점수를 계산하라"는 책임
→ 카드 목록을 가진 객체(Hand)가 정보 전문가
→ 각 카드의 점수를 아는 Card가 정보 전문가
```

### 5.2 CREATOR (창조자)

- 다음 조건을 가장 많이 만족하는 객체 B에게 객체 A 생성 책임을 할당한다:
    - B가 A를 포함하거나 참조한다
    - B가 A를 기록한다
    - B가 A를 긴밀하게 사용한다
    - B가 A를 초기화하는 데 필요한 데이터를 가지고 있다

### 5.3 LOW COUPLING (낮은 결합도)

- 설계의 전체적인 결합도가 낮게 유지되도록 책임을 할당한다.
- 의존성을 낮추고 변화의 영향을 줄이고 재사용성을 증가시킨다.

### 5.4 HIGH COHESION (높은 응집도)

- 높은 응집도를 유지할 수 있게 책임을 할당한다.
- 복잡성을 관리할 수 있는 수준으로 유지한다.

### 5.5 POLYMORPHISM (다형성)

- 타입에 따라 변하는 행동이 있다면 타입을 분리하고 변화하는 행동을 각 타입의 책임으로 할당한다.
- if-else나 switch 문을 사용해서 타입을 확인하는 코드 대신 다형성을 이용한다.

### 5.6 PROTECTED VARIATIONS (변경 보호)

- 변화가 예상되는 불안정한 지점들을 식별하고, 그 주위에 안정된 인터페이스를 형성하도록 책임을 할당한다.
- 변경될 가능성이 높은 부분을 별도의 클래스로 분리하고, 이에 대한 인터페이스를 만들어 변경의 영향을 차단한다.

## 6장. 메시지와 인터페이스

### 6.1 협력과 메시지

- **메시지**: 객체들이 협력하기 위해 사용하는 유일한 의사소통 수단.
- 메시지 전송 = 오퍼레이션명 + 인자
- **메서드**: 메시지를 수신했을 때 실제로 실행되는 함수 또는 프로시저
- 메시지와 메서드를 구분하는 것이 다형성의 핵심이다.

### 6.2 인터페이스와 설계 품질

좋은 인터페이스는 다음과 같은 특성을 가진다:

- **최소한의 인터페이스**: 꼭 필요한 오퍼레이션만을 포함한다.
- **추상적인 인터페이스**: 어떻게(how)가 아니라 무엇(what)을 하는지를 표현한다.

### 6.3 디미터 법칙 (Law of Demeter)

- 객체의 내부 구조에 강하게 결합되지 않도록 협력 경로를 제한한다.
- "낯선 자에게 말하지 말라" 또는 "오직 인접한 이웃하고만 말하라"
- 클래스 C의 메서드 M은 다음과 같은 객체의 메서드만 호출해야 한다:
    - 클래스 C 자체
    - M에 의해 생성된 객체
    - M의 인자로 전달된 객체
    - C의 인스턴스 변수가 참조하는 객체

```java
// 디미터 법칙 위반
participant.getHand().

getCards().

stream()...

// 디미터 법칙 준수
        participant.

calculateScore()
```

### 6.4 묻지 말고 시켜라 (Tell, Don't Ask)

- 객체의 상태를 묻는 오퍼레이션을 호출한 후 반환된 상태를 기반으로 결정을 내리지 않는다.
- 대신 원하는 것을 시키고, 객체 스스로 판단하게 한다.
- 상태를 묻는 오퍼레이션 → 행동을 요청하는 오퍼레이션으로 대체.

```java
// 묻고 있는 코드 (나쁜 예)
if(participant.getScore() >21){
        participant.

setBust(true);
}

// 시키는 코드 (좋은 예)
        participant.

checkBust();
```

### 6.5 의도를 드러내는 인터페이스

- 메서드의 이름은 **어떻게 수행하느냐**가 아니라 **무엇을 하느냐**에 따라 짓는다.
- 클라이언트의 의도를 드러내야 한다.

```java
// 어떻게(how) - 나쁜 예
public void addCardToHand(Card card) { ...}

// 무엇(what) - 좋은 예
public void receive(Card card) { ...}
```

### 6.6 명령-쿼리 분리 원칙 (Command-Query Separation)

- **명령(Command)**: 객체의 상태를 수정하는 오퍼레이션. 반환값 없음.
- **쿼리(Query)**: 객체의 정보를 반환하는 오퍼레이션. 상태 변경 없음.
- 어떤 오퍼레이션도 명령인 동시에 쿼리여서는 안 된다.
- 명령과 쿼리를 분리하면 코드를 예측하고 이해하기 쉬워진다.

## 7장. 객체 분해

### 7.1 추상화의 종류

- **프로시저 추상화**: 기능 분해(알고리즘 분해)
- **데이터 추상화**: 데이터 중심으로 타입을 추상화(추상 데이터 타입) 또는 데이터 중심으로 프로시저를 추상화(객체지향)
- 프로시저 추상화를 중심으로 한 하향식 기능 분해는 변경에 취약하다.

### 7.2 추상 데이터 타입 vs 객체지향

- **추상 데이터 타입**: 오퍼레이션을 기준으로 타입을 묶는다. 타입을 추상화.
- **객체지향**: 타입을 기준으로 오퍼레이션을 묶는다. 절차를 추상화.
- 추상 데이터 타입은 타입 추가 시 기존 코드를 수정해야 한다.
- 객체지향은 새로운 타입 추가가 용이하다 (다형성).

## 8장. 의존성 관리하기

### 8.1 의존성 이해하기

- 의존성은 **방향**이 있다: A가 B에 의존하면 B가 변경될 때 A가 영향을 받는다.
- 의존성은 **전이(transitive)**될 수 있다: A → B → C이면 A → C
- 직접 의존성 vs 간접 의존성을 구분한다.

### 8.2 런타임 의존성과 컴파일타임 의존성

- 런타임 의존성: 실행 시점에 객체 사이의 의존성
- 컴파일타임 의존성: 코드에 드러나는 클래스 사이의 의존성
- **유연하고 재사용 가능한 설계를 위해서는 동일한 소스코드 구조를 가지고 다양한 실행 구조를 만들 수 있어야 한다.**

### 8.3 의존성 해결 방법

1. **생성자를 통한 의존성 해결**: 객체를 생성하는 시점에 의존성 해결
2. **setter 메서드를 통한 의존성 해결**: 객체 생성 후 의존성 해결 (실행 시점에 변경 가능)
3. **메서드 인자를 통한 의존성 해결**: 메서드 실행 시마다 의존성 해결

### 8.4 유연한 설계

#### 의존성과 결합도

- 의존성 자체는 나쁜 것이 아니다. 단지 **결합도의 정도**가 문제다.
- 바람직한 의존성: 재사용성을 해치지 않는 의존성 → **느슨한 결합도(loose coupling)**
- 바람직하지 못한 의존성: 재사용성을 해치는 의존성 → **강한 결합도(tight coupling)**

#### 추상화에 의존하라

- 의존성의 대상을 다음과 같이 구분할 수 있다 (아래로 갈수록 결합도가 낮음):
    1. 구체 클래스 의존성 (가장 높은 결합도)
    2. 추상 클래스 의존성
    3. 인터페이스 의존성 (가장 낮은 결합도)
- 구체 클래스가 아닌 추상화에 의존하면 결합도를 낮출 수 있다.

#### new는 해롭다

- `new`를 사용하면 구체 클래스에 대한 결합도가 높아진다.
- `new`는 구체 클래스의 이름을 직접 기술해야 하고, 생성자에 전달하는 인자도 알아야 한다.
- 해결법: 인스턴스 생성 로직과 사용 로직을 분리한다.
- 사용하는 객체가 아닌 외부의 객체가 인스턴스를 생성하여 전달한다 (의존성 주입).

```java
// new가 해로운 예
public class BlackjackGame {
    public BlackjackGame() {
        this.deck = new Deck(Card.createFullDeck()); // 구체 클래스에 강하게 결합
    }
}

// 의존성 주입
public class BlackjackGame {
    public BlackjackGame(Deck deck) { // 외부에서 주입
        this.deck = deck;
    }
}
```

## 9장. 유연한 설계

### 9.1 개방-폐쇄 원칙 (OCP)

- 소프트웨어 개체는 **확장에 대해 열려** 있어야 하고, **수정에 대해 닫혀** 있어야 한다.
- 추상화에 의존함으로써 OCP를 따르는 설계를 할 수 있다.
- 컴파일타임 의존성을 고정시키고 런타임 의존성을 변경한다.

### 9.2 생성 사용 분리 (Separating Use from Creation)

- 소프트웨어 시스템은 **시작 단계**(객체를 생성하는 단계)와 **실행 단계**(객체를 사용하는 단계)를 분리해야 한다.
- FACTORY: 생성과 사용을 분리하기 위해 객체 생성에 특화된 객체를 사용한다.
- PURE FABRICATION: 도메인 개념을 표현한 것이 아니라 순수하게 기술적인 결정에 의해 추가된 객체.

### 9.3 의존성 주입 (Dependency Injection)

- 사용하는 객체가 아닌 외부의 독립적인 객체가 인스턴스를 생성한 후 이를 전달해서 의존성을 해결하는 방법.
- **생성자 주입**: 가장 일반적이고 권장되는 방법.
- **setter 주입**: 의존 대상을 변경할 필요가 있을 때.
- **메서드 주입**: 메서드 호출 시점에 의존 대상이 달라질 때.

### 9.4 의존성 역전 원칙 (DIP)

1. 상위 수준의 모듈은 하위 수준의 모듈에 의존해서는 안 된다. 둘 모두 추상화에 의존해야 한다.
2. 추상화는 구체적인 사항에 의존해서는 안 된다. 구체적인 사항은 추상화에 의존해야 한다.

## 10장. 상속과 코드 재사용

### 10.1 상속의 문제점

#### 불필요한 인터페이스 상속 문제

- 자식 클래스에게 부적합한 부모 클래스의 오퍼레이션이 상속될 수 있다.
- 예: `Stack`이 `Vector`를 상속받으면 `add(index, element)` 같은 부적합한 메서드까지 상속됨.

#### 메서드 오버라이딩의 오작용 문제

- 자식 클래스가 부모 클래스의 메서드를 오버라이딩할 때, 부모 클래스의 내부 구현을 정확하게 알아야 한다.
- 이는 캡슐화를 위반한다.

#### 부모 클래스와 자식 클래스의 동시 수정 문제

- 자식 클래스가 부모 클래스의 구현에 강하게 결합되어 있으면, 부모 클래스를 변경할 때 자식 클래스도 함께 변경해야 할 수 있다.

### 10.2 추상화를 이용한 해결

- 부모 클래스가 제공하는 모든 것이 자식 클래스에게 적합한지 확인한다.
- 상속을 사용할 때는 **추상화에 의존**하도록 한다.
- 두 메서드가 유사하게 보인다면 **차이점을 메서드로 추출**하고, 추출한 메서드의 공통 부분을 부모 클래스로 올린다.

## 11장. 합성과 유연한 설계

### 11.1 상속 vs 합성

| 상속                | 합성               |
|-------------------|------------------|
| is-a 관계           | has-a 관계         |
| 부모 클래스의 코드를 재사용   | 객체의 인터페이스를 재사용   |
| 화이트박스 재사용 (내부 공개) | 블랙박스 재사용 (내부 감춤) |
| 컴파일타임에 관계 결정      | 런타임에 관계 결정 가능    |
| 부모 클래스에 강하게 결합    | 느슨한 결합 유지        |

### 11.2 합성을 사용하라

- 상속은 부모 클래스 안에 구현된 코드 자체를 재사용한다.
- 합성은 포함되는 객체의 **퍼블릭 인터페이스**를 재사용한다.
- 따라서 합성을 이용하면 구현에 대한 의존성을 인터페이스에 대한 의존성으로 변경할 수 있다.
- **코드 재사용이 목적이라면 상속보다 합성을 선호하라.**

### 11.3 믹스인 (Mixin)

- 특정한 행동을 클래스에 섞어 넣는 방법.
- 상속과 합성의 중간 형태.
- Java에서는 인터페이스의 디폴트 메서드가 제한적인 믹스인 역할을 한다.

## 12장. 다형성

### 12.1 다형성의 종류

- **유니버설 다형성**: 매개변수 다형성(제네릭), 포함 다형성(서브타입 다형성)
- **임시 다형성**: 오버로딩 다형성, 강제 다형성

### 12.2 상속의 양면성

- **데이터 관점의 상속**: 부모 클래스에서 정의한 모든 데이터를 자식 클래스의 인스턴스에 자동으로 포함
- **행동 관점의 상속**: 부모 클래스에서 정의한 모든 행동을 자식 클래스에서 자동으로 수행
- 행동 관점에서 상속은 부모 클래스가 정의한 것뿐만 아니라 부모 클래스가 정의한 메서드를 자식이 실행할 수 있게 한다.

## 13장. 서브클래싱과 서브타이핑

### 13.1 타입과 서브타이핑

- **서브클래싱 (구현 상속)**: 코드 재사용 목적. is-a 관계가 성립하지 않을 수 있음.
- **서브타이핑 (인터페이스 상속)**: 타입 계층을 구성하기 위한 상속. 대체 가능성이 핵심.

### 13.2 리스코프 치환 원칙 (LSP)

- "서브타입은 그것의 기반 타입에 대해 대체 가능해야 한다."
- 클라이언트가 차이점을 인식하지 못한 채 파생 클래스의 인터페이스를 통해 서브클래스를 사용할 수 있어야 한다.
- LSP를 위반하면 서브타입이 아니다 — 상속 관계가 아무리 자연스러워 보여도.

### 13.3 계약에 의한 설계 (Design by Contract)

- **사전 조건(precondition)**: 메서드가 호출되기 위해 만족되어야 하는 조건.
    - 서브타입에서 사전 조건을 강화할 수 없다.
- **사후 조건(postcondition)**: 메서드 실행 후 보장되어야 하는 조건.
    - 서브타입에서 사후 조건을 약화할 수 없다.
- **클래스 불변식(class invariant)**: 인스턴스 생명주기 전반에 걸쳐 항상 참이어야 하는 조건.
    - 서브타입에서 불변식은 유지되어야 한다.
