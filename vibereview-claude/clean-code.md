# Clean Code (Robert C. Martin) 가이드라인

> 코드 리뷰 시 참고할 Clean Code의 핵심 원칙들을 정리한 문서입니다.

---

## 1. 의미 있는 이름 (Meaningful Names)

### 1.1 의도를 분명히 밝혀라

- 변수, 함수, 클래스 이름만으로 존재 이유, 수행 기능, 사용 방법을 알 수 있어야 한다.
- 이름을 지을 때 "이 변수가 무엇을 의미하는가?"라는 질문에 답할 수 있어야 한다.

```java
// 나쁜 예
int d; // 경과 시간(날짜)

// 좋은 예
int elapsedTimeInDays;
```

### 1.2 그릇된 정보를 피하라

- 널리 쓰이는 의미가 있는 단어를 다른 의미로 사용하지 않는다.
- 서로 흡사한 이름을 사용하지 않는다.
- `accountList`라고 이름 짓고 실제 타입이 List가 아니면 그릇된 정보다.

### 1.3 의미 있게 구분하라

- 연속된 숫자를 덧붙이거나(a1, a2, ...) 불용어(noise word)를 추가하는 방식은 피한다.
- `ProductInfo`와 `ProductData`는 의미 있는 구분이 아니다.
- `nameString`에서 `String`은 불용어다.

### 1.4 발음하기 쉬운 이름을 사용하라

- 발음할 수 없는 이름은 토론할 수 없다.

### 1.5 검색하기 쉬운 이름을 사용하라

- 한 글자 이름이나 상수는 코드에서 쉽게 검색할 수 없다.
- 이름 길이는 범위(scope) 크기에 비례해야 한다.
- 로컬 변수만 한 문자 변수를 사용한다.

### 1.6 인코딩을 피하라

- 헝가리안 표기법을 사용하지 않는다.
- 멤버 변수 접두어(`m_`)를 사용하지 않는다.
- 인터페이스 이름에 접두어 `I`를 붙이지 않는다.

### 1.7 클래스 이름

- 명사 또는 명사구를 사용한다: `Customer`, `WikiPage`, `Account`
- `Manager`, `Processor`, `Data`, `Info` 같은 모호한 단어는 피한다.
- 동사를 사용하지 않는다.

### 1.8 메서드 이름

- 동사 또는 동사구를 사용한다: `postPayment`, `deletePage`, `save`
- 접근자(Accessor)는 `get`, 변경자(Mutator)는 `set`, 조건자(Predicate)는 `is`를 접두어로 붙인다.
- 생성자를 오버로딩할 때는 정적 팩토리 메서드를 사용한다.

### 1.9 한 개념에 한 단어를 사용하라

- 추상적인 개념 하나에 단어 하나를 선택하고 일관적으로 사용한다.
- `fetch`, `retrieve`, `get`을 혼용하지 않는다.
- `controller`, `manager`, `driver`를 섞어 쓰지 않는다.

### 1.10 해법 영역에서 가져온 이름을 사용하라

- 기술 개념에는 기술 이름이 가장 적합하다.
- `AccountVisitor`, `JobQueue` 등 프로그래머에게 익숙한 패턴 이름을 활용한다.

### 1.11 문제 영역에서 가져온 이름을 사용하라

- 적절한 프로그래머 용어가 없다면, 문제 영역(도메인)에서 이름을 가져온다.

## 2. 함수 (Functions)

### 2.1 작게 만들어라

- 함수는 가능한 한 작아야 한다.
- if문/else문/while문 등에 들어가는 블록은 한 줄이어야 한다 (함수 호출).
- 들여쓰기(indent) 수준은 1단이나 2단을 넘어서지 않는다.

### 2.2 한 가지만 해라

- 함수는 한 가지를 해야 한다. 그 한 가지를 잘 해야 한다. 그 한 가지만을 해야 한다.
- 함수가 한 가지만 하는지 판단하는 방법: 의미 있는 이름으로 다른 함수를 추출할 수 있다면, 그 함수는 여러 작업을 하는 것이다.

### 2.3 함수 당 추상화 수준은 하나로

- 한 함수 내에 추상화 수준을 섞지 않는다.
- 높은 수준의 개념과 낮은 수준의 세부 구현이 한 함수에 공존하면 안 된다.
- **내려가기 규칙**: 코드는 위에서 아래로 이야기처럼 읽혀야 한다.

### 2.4 Switch 문

- switch 문은 본질적으로 N가지 일을 한다.
- 다형성을 이용해 switch 문을 저차원 클래스에 숨기고, 반복하지 않는다.
- 추상 팩토리 안에서 사용하여 적절한 파생 클래스 인스턴스를 생성하는 것은 허용.

### 2.5 서술적인 이름을 사용하라

- 길고 서술적인 이름이 짧고 어려운 이름보다 좋다.
- 길고 서술적인 이름이 서술적인 긴 주석보다 좋다.
- 이름을 붙일 때는 일관성이 있어야 한다.

### 2.6 함수 인수

- 이상적인 인수 개수는 0개(무항)이다.
- 다음은 1개(단항), 다음은 2개(이항)이다.
- 3개(삼항)는 가능한 피한다. 4개 이상(다항)은 특별한 이유가 있어도 사용하면 안 된다.
- 많은 인수가 필요하면 독자적인 클래스로 묶는 것을 고려한다.

```java
// 나쁜 예
Circle makeCircle(double x, double y, double radius);

// 좋은 예
Circle makeCircle(Point center, double radius);
```

### 2.7 부수 효과를 일으키지 마라

- 함수 이름에서 드러나지 않는 부수 효과를 만들지 않는다.
- 시간적인 결합(temporal coupling)을 초래하는 부수 효과를 피한다.
- 출력 인수를 사용하지 않는다. 객체 상태를 변경해야 한다면 해당 객체의 메서드를 호출한다.

### 2.8 명령과 조회를 분리하라

- 함수는 뭔가를 수행하거나(명령) 뭔가에 답하거나(조회) 둘 중 하나만 해야 한다.
- 객체 상태를 변경하거나 객체 정보를 반환하거나 둘 중 하나다.

```java
// 나쁜 예 - 명령과 조회가 혼합
if(set("username","unclebob"))...

// 좋은 예 - 분리
        if(

attributeExists("username")){

setAttribute("username","unclebob");
}
```

### 2.9 오류 코드보다 예외를 사용하라

- 오류 코드를 반환하면 호출자에게 즉시 처리를 강요한다.
- 예외를 사용하면 오류 처리 코드가 원래 코드에서 분리된다.
- try/catch 블록은 별도 함수로 뽑아낸다.
- 오류 처리도 한 가지 작업이다.

### 2.10 반복하지 마라 (DRY)

- 중복은 소프트웨어에서 모든 악의 근원이다.
- 중복을 발견하면 함수로 추출한다.

## 3. 주석 (Comments)

### 3.1 주석은 나쁜 코드를 보완하지 못한다

- 코드로 의도를 표현하라.
- 주석이 필요한 상황이 오면, 코드를 리팩토링하여 주석이 필요 없게 만들어라.

### 3.2 좋은 주석

- **법적인 주석**: 저작권 정보, 라이선스 정보
- **정보를 제공하는 주석**: 정규표현식 패턴의 의미 설명 등
- **의도를 설명하는 주석**: 구현을 이해하게 도와주는 것을 넘어 결정에 깔린 의도를 설명
- **의미를 명료하게 밝히는 주석**: 인수나 반환값이 표준 라이브러리이거나 변경하지 못하는 코드일 때
- **결과를 경고하는 주석**: 다른 프로그래머에게 결과를 경고
- **TODO 주석**: 앞으로 할 일을 남겨둠
- **중요성을 강조하는 주석**: 대수롭지 않다고 여겨질 수 있는 것의 중요성을 강조

### 3.3 나쁜 주석

- **주절거리는 주석**: 의무감으로 다는 주석
- **같은 이야기를 중복하는 주석**: 코드 내용을 그대로 반복
- **오해할 여지가 있는 주석**: 부정확한 정보
- **의무적으로 다는 주석**: 모든 함수에 Javadoc을 강제하는 규칙
- **이력을 기록하는 주석**: 소스 코드 관리 시스템이 대신한다
- **있으나 마나 한 주석**: 너무 당연한 사실
- **닫는 괄호에 다는 주석**: 함수를 줄여라
- **주석으로 처리한 코드**: 소스 코드 관리 시스템이 기억한다. 그냥 삭제하라.
- **HTML 주석**: 주석에 HTML 태그를 삽입하지 않는다
- **전역 정보**: 근처 코드만 기술하라
- **비공개 코드에서 Javadoc**: 공개 API가 아니라면 불필요

## 4. 형식 맞추기 (Formatting)

### 4.1 적절한 행 길이를 유지하라

- 200줄 미만의 파일로도 커다란 시스템을 구축할 수 있다.
- 작은 파일이 이해하기 쉽다.

### 4.2 신문 기사처럼 작성하라

- 이름은 간단하면서도 설명이 가능하게 짓는다.
- 소스 파일 첫 부분은 고차원 개념과 알고리즘을 설명한다.
- 아래로 내려갈수록 의도를 세세하게 묘사한다.
- 마지막에는 가장 저차원 함수와 세부 내역이 나온다.

### 4.3 세로 밀집도

- 서로 밀접한 코드 행은 세로로 가까이 놓는다.
- 연관성이 깊은 두 개념은 세로 거리로 연관성을 표현한다.

### 4.4 변수 선언

- 변수는 사용하는 위치에 최대한 가까이 선언한다.
- 루프를 제어하는 변수는 루프 문 내부에 선언한다.

### 4.5 인스턴스 변수

- 인스턴스 변수는 클래스 맨 처음에 선언한다.
- 잘 알려진 위치에 인스턴스 변수를 모은다.

### 4.6 종속 함수

- 한 함수가 다른 함수를 호출한다면 두 함수는 세로로 가까이 배치한다.
- 호출하는 함수를 호출되는 함수보다 먼저 배치한다.

### 4.7 가로 형식 맞추기

- 프로그래머는 짧은 행을 선호한다.
- 120자 정도를 행 길이 제한으로 제안한다.

## 5. 객체와 자료 구조 (Objects and Data Structures)

### 5.1 자료 추상화

- 구현을 감추려면 추상화가 필요하다.
- 변수 사이에 함수라는 계층을 넣는다고 구현이 저절로 감춰지지는 않는다.
- 추상 인터페이스를 제공해 사용자가 구현을 모른 채 자료의 핵심을 조작할 수 있어야 한다.

```java
// 구체적인 Vehicle - 구현을 노출
public interface Vehicle {
    double getFuelTankCapacityInGallons();

    double getGallonsOfGasoline();
}

// 추상적인 Vehicle - 핵심을 추상화
public interface Vehicle {
    double getPercentFuelRemaining();
}
```

### 5.2 자료/객체 비대칭

- **객체**: 추상화 뒤로 자료를 숨기고, 자료를 다루는 함수만 공개한다.
- **자료 구조**: 자료를 그대로 공개하며, 별다른 함수는 제공하지 않는다.
- 새로운 자료 타입이 필요하면 → 객체 지향이 적합
- 새로운 함수가 필요하면 → 절차적 코드와 자료 구조가 적합

### 5.3 디미터 법칙 (Law of Demeter)

- 모듈은 자신이 조작하는 객체의 내부를 몰라야 한다.
- 객체는 자료를 숨기고 함수를 공개한다.
- **기차 충돌(Train Wreck)**을 피하라:

```java
// 나쁜 예 - 기차 충돌
final String outputDir = ctxt.getOptions().getScratchDir().getAbsolutePath();

// 좋은 예 - 분리
Options opts = ctxt.getOptions();
File scratchDir = opts.getScratchDir();
final String outputDir = scratchDir.getAbsolutePath();
```

### 5.4 DTO (Data Transfer Object)

- 자료 구조체의 전형적인 형태는 공개 변수만 있고 함수가 없는 클래스다.
- Bean 구조: 비공개 변수를 getter/setter로 조작. 별다른 이익 없음.
- **활성 레코드**: DTO의 특수한 형태. 비즈니스 규칙 메서드를 추가하지 않는다.

## 6. 오류 처리 (Error Handling)

### 6.1 오류 코드보다 예외를 사용하라

- 오류 코드는 호출자 코드를 복잡하게 만든다.
- 예외를 사용하면 논리와 오류 처리가 뒤섞이지 않는다.

### 6.2 Try-Catch-Finally 문부터 작성하라

- 예외가 발생할 코드를 짤 때는 try-catch-finally 문으로 시작하라.
- 호출자가 기대하는 상태를 정의하기 쉬워진다.

### 6.3 미확인(unchecked) 예외를 사용하라

- 확인된(checked) 예외는 OCP를 위반한다.
- 하위 단계에서 확인된 예외를 추가하면 상위 단계 메서드 선언부를 전부 고쳐야 한다.
- 일반적인 애플리케이션에서는 확인된 예외의 의존성 비용이 이익보다 크다.

### 6.4 예외에 의미를 제공하라

- 예외를 던질 때 전후 상황을 충분히 덧붙인다.
- 오류 메시지에 실패한 연산 이름과 실패 유형을 언급한다.

### 6.5 호출자를 고려해 예외 클래스를 정의하라

- 오류를 분류하는 방법 중 가장 중요한 것은 오류를 잡아내는 방법이다.
- 외부 API를 사용할 때는 감싸기(wrapper) 클래스로 예외를 통합한다.

### 6.6 정상 흐름을 정의하라

- 특수 사례 패턴(Special Case Pattern): 클래스를 만들거나 객체를 조작해 특수 사례를 처리한다.
- 클래스나 객체가 예외적인 상황을 캡슐화하면 클라이언트 코드가 예외를 처리할 필요가 없어진다.

### 6.7 null을 반환하지 마라

- null을 반환하는 코드는 호출자에게 null 확인 작업을 떠넘긴다.
- null 대신 예외를 던지거나 특수 사례 객체를 반환한다.
- 빈 리스트를 반환하면 `Collections.emptyList()`로 null 반환을 피할 수 있다.

### 6.8 null을 전달하지 마라

- 정상적인 인수로 null을 기대하는 API가 아니라면 메서드로 null을 전달하지 않는다.
- null 인수를 방어하는 코드보다 애초에 null을 넘기지 못하게 하는 정책이 합리적이다.

## 7. 경계 (Boundaries)

### 7.1 외부 코드를 감싸라

- 경계 인터페이스(Map, 외부 API 등)를 이용할 때는 클래스 안으로 숨긴다.
- 경계 인터페이스를 여기저기 넘기지 않는다.

### 7.2 학습 테스트를 작성하라

- 외부 코드를 익히기 위해 학습 테스트를 작성한다.
- 통제된 실험으로 외부 API를 이해한다.
- 외부 패키지가 변경되었을 때 학습 테스트가 호환성을 검증한다.

## 8. 단위 테스트 (Unit Tests)

### 8.1 TDD의 세 가지 법칙

1. 실패하는 단위 테스트를 작성할 때까지 실제 코드를 작성하지 않는다.
2. 컴파일은 실패하지 않으면서 실행이 실패하는 정도로만 단위 테스트를 작성한다.
3. 현재 실패하는 테스트를 통과할 정도로만 실제 코드를 작성한다.

### 8.2 깨끗한 테스트 코드

- 테스트 코드는 실제 코드 못지않게 중요하다.
- 테스트 코드는 실제 코드 못지않게 깨끗하게 짠다.
- 가독성이 가장 중요하다: 명료성, 단순성, 풍부한 표현력.

### 8.3 테스트 당 assert 하나

- assert 문이 단 하나인 함수는 결론이 하나라서 이해하기 쉽다.
- 최소한 개념 하나만 테스트하라.

### 8.4 F.I.R.S.T 규칙

- **Fast**: 테스트는 빨리 돌아야 한다.
- **Independent**: 각 테스트는 서로 의존하면 안 된다.
- **Repeatable**: 어떤 환경에서도 반복 가능해야 한다.
- **Self-Validating**: 부울(bool) 값으로 결과를 내야 한다 (성공/실패).
- **Timely**: 실제 코드를 구현하기 직전에 구현한다.

### 8.5 테스트 당 개념 하나

- 테스트 함수 하나는 개념 하나만 테스트한다.
- 이것저것 잡다한 개념을 연속으로 테스트하는 긴 함수는 피한다.

## 9. 클래스 (Classes)

### 9.1 클래스는 작아야 한다

- 함수처럼 클래스도 작아야 한다.
- 클래스의 크기는 맡은 **책임**의 수로 측정한다.
- 클래스 이름은 해당 클래스의 책임을 기술해야 한다.
- 간결한 이름이 떠오르지 않으면 클래스 크기가 너무 큰 것이다.
- 클래스 설명은 "if", "and", "or", "but"을 사용하지 않고 25단어 내외로 가능해야 한다.

### 9.2 단일 책임 원칙 (SRP)

- 클래스나 모듈을 변경할 이유가 하나, 단 하나뿐이어야 한다.
- 변경할 이유를 파악하려 애쓰다 보면 코드를 추상화하기도 쉬워진다.
- 큰 클래스 몇 개가 아니라 작은 클래스 여럿으로 이뤄진 시스템이 더 바람직하다.

### 9.3 응집도 (Cohesion)

- 클래스는 인스턴스 변수 수가 적어야 한다.
- 각 클래스 메서드는 클래스 인스턴스 변수를 하나 이상 사용해야 한다.
- 모든 인스턴스 변수를 메서드마다 사용하는 클래스는 응집도가 가장 높다.
- 응집도가 높다는 것은 클래스에 속한 메서드와 변수가 서로 의존하며 논리적인 단위로 묶인다는 뜻이다.

### 9.4 변경하기 쉬운 클래스

- 요구사항은 변하기 마련이므로, 코드도 변하기 마련이다.
- 변경으로부터 격리하려면 추상화에 의존한다.
- OCP(Open-Closed Principle): 확장에 개방적이고 수정에 폐쇄적이어야 한다.
- DIP(Dependency Inversion Principle): 클래스가 상세한 구현이 아니라 추상화에 의존해야 한다.

## 10. 창발성 (Emergence) — 켄트 벡의 단순한 설계 규칙

### 10.1 모든 테스트를 실행하라

- 테스트가 가능한 시스템을 만들면 설계 품질이 높아진다.
- SRP를 준수하는 클래스는 테스트가 쉽다.
- 결합도가 낮으면 테스트가 쉽다.

### 10.2 중복을 없애라

- 깔끔한 시스템을 만들려면 단 몇 줄이라도 중복을 제거한다.
- 소규모 재사용은 시스템 복잡도를 극적으로 줄여준다.
- TEMPLATE METHOD 패턴은 고차원 중복을 제거하는 좋은 기법이다.

### 10.3 프로그래머 의도를 표현하라

- 코드는 개발자의 의도를 분명히 표현해야 한다.
- 좋은 이름을 선택한다.
- 함수와 클래스 크기를 가능한 줄인다.
- 표준 명칭을 사용한다 (디자인 패턴 등).
- 단위 테스트를 꼼꼼히 작성한다.

### 10.4 클래스와 메서드 수를 최소로 줄여라

- 중복을 제거하고, 의도를 표현하고, SRP를 준수한다는 기본 개념도 극단으로 치달으면 득보다 실이 많다.
- 함수와 클래스 수를 가능한 줄이되, 이 규칙은 위 세 규칙보다 우선순위가 낮다.
