# 🚀 코드 품질 평가 가이드라인 (Unified Standards)

이 문서는 **Clean Code**, **Effective Java**, **오브젝트**의 핵심 원칙을 통합하여, 블랙잭 미션 수행 및 코드 리뷰 시 준수해야 할 객관적인 코드 품질 기준을 제시합니다.

---

## 1. 객체 지향 및 설계 (Object-Oriented Design)

### 1.1 책임 중심 설계 (Responsibility-Driven Design)

* **데이터보다 행동 우선:** "이 객체가 어떤 데이터를 가져야 하는가?"가 아니라 **"이 객체가 어떤 책임을 져야 하는가?"**를 먼저 결정한다. (오브젝트 4장)
* **정보 전문가(Information Expert) 패턴:** 책임을 수행하는 데 필요한 정보를 가장 많이 알고 있는 객체에게 책임을 할당한다. (오브젝트 5장)
* **단일 책임 원칙 (SRP):** 클래스와 메서드는 단 하나의 변경 이유만을 가져야 하며, 한 가지 일만 수행해야 한다. (Clean Code 2, 9장)

### 1.2 자율적인 객체와 협력

* **묻지 말고 시켜라 (Tell, Don't Ask):** 객체의 상태를 getter로 물어보고 외부에서 판단하지 말고, 객체에게 행동을 위임한다. (오브젝트 6장)
* **디미터 법칙 (Law of Demeter):** 낯선 자에게 말하지 말라. `a.getB().getC().doSomething()`과 같은 기차 충돌 코드를 지양하고, 인접한 이웃하고만 메시지를
  주고받는다. (Clean Code 5장, 오브젝트 6장)
* **캡슐화 준수:** 필드는 `private`으로 유지하고, 외부에는 최소한의 인터페이스만 노출한다. 내부 구현의 변경이 외부에 파급되지 않도록 격리한다. (오브젝트 1, 4장)

## 2. 가독성 및 의사소통 (Readability & Communication)

### 2.1 의미 있는 네이밍

* **의도 노출:** 변수, 함수, 클래스 이름은 존재 이유와 기능을 명확히 드러내야 한다. 축약어를 지양하고 도메인 용어를 적극 활용한다. (Clean Code 1장)
* **추상화 수준에 맞는 이름:** "어떻게(How)"가 아니라 **"무엇(What)"**을 하는지 나타내는 의도를 드러내는 이름을 짓는다. (오브젝트 6장)

### 2.2 함수 및 클래스 구조

* **작게 만들기:** 함수는 물리적으로 짧아야 하며(최대한 10줄 내외), 클래스 역시 책임의 수에 따라 작게 유지한다. (Clean Code 2, 9장)
* **내려가기 규칙:** 코드는 위에서 아래로 이야기처럼 자연스럽게 읽혀야 하며, 한 함수 내의 추상화 수준은 동일해야 한다. (Clean Code 2장)

## 3. Java 언어 특성 및 안정성 (Effective Java)

### 3.1 객체 생성과 상태 관리

* **정적 팩토리 메서드:** 생성자 대신 의도가 명확한 이름을 가진 정적 팩토리 메서드(`of`, `from`) 사용을 고려한다. (Effective Java 아이템 1)
* **변경 가능성 최소화 (불변 객체):** 클래스는 꼭 필요한 경우가 아니면 불변(`final` 필드 활용)이어야 한다. 가변 상태가 적을수록 객체는 안전하고 예측 가능하다. (Effective Java 아이템
    17)
* **상속보다는 합성:** 상속은 캡슐화를 깨뜨리고 강한 결합을 만든다. `is-a` 관계가 확실하지 않다면 컴포지션(Composition)을 우선한다. (Effective Java 아이템 18, 오브젝트 11장)

### 3.2 방어적 프로그래밍

* **매개변수 유효성 검사:** 메서드 몸체가 실행되기 전에 인수를 검사한다. (`Objects.requireNonNull` 등 활용) (Effective Java 아이템 49)
* **방어적 복사:** 내부 컬렉션이나 가변 객체를 반환할 때 외부에서 조작하지 못하도록 방어적 복사본을 반환한다. (`Collections.unmodifiableList` 등) (Effective Java 아이템
    50)
* **null 대신 빈 컨테이너:** null을 반환하거나 전달하지 않는다. 대신 빈 컬렉션이나 `Optional`을 사용한다. (Effective Java 아이템 54, Clean Code 6장)

## 4. 예외 처리 및 오류 관리

* **오류 코드보다 예외:** 비즈니스 로직과 오류 처리 코드를 분리하기 위해 예외를 사용한다. (Clean Code 6장)
* **표준 예외 활용:** `IllegalArgumentException`, `IllegalStateException` 등 Java 표준 예외를 상황에 맞게 재사용한다. (Effective Java 아이템 72)
* **Checked vs Unchecked:** 호출자가 복구할 수 있는 경우에만 Checked Exception을 고려하되, 일반적으로는 OCP를 준수하기 위해 Unchecked(Runtime)
  Exception을 선호한다. (Clean Code 6장, Effective Java 아이템 70)

---

## 5. 테스트 코드 품질 (Unit Testing)

* **F.I.R.S.T 원칙:** Fast(빠름), Independent(독립적), Repeatable(반복 가능), Self-Validating(자가 검증), Timely(적시성). (Clean Code 8장)
* **테스트 당 하나의 개념:** 테스트 함수 하나는 한 가지 개념만 검증해야 하며, `given-when-then` 구조로 가독성을 높인다. (Clean Code 8장)
* **경계 조건 검증:** 정상 흐름뿐만 아니라 경계값(Boundary values), 예외 케이스를 철저히 검증한다.
